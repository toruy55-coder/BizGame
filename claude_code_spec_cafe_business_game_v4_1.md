## 1. v4.1での重要変更点

### 1.1 ゲーム期間

- 7日間で終了する。
- 月曜日から開始する。
- 6日目・7日目は土日として扱う。
- 6日目・7日目は学園祭イベントとする。

### 1.2 履歴画面の追加

- 過去の営業結果を確認できる「履歴画面」を追加する。
- 履歴画面は別タブ、または画面内タブで表示してよい。
- 過去結果は確認のみでよい。過去入力の編集は不要。

### 1.3 商品変更

販売商品は以下の3種類にする。

- コーヒー
- サンドイッチ
- クッキー

以下は廃止する。

- いちごトッピング
- 焼き菓子という商品名

焼き菓子に相当する商品は、**クッキー**として扱う。

### 1.4 SNS仕様の修正

6日目の学園祭1日目に、過去のSNS投稿状況を反映する。

#### 残す仕様

```text
6日目に、これまでSNSを4日以上発信していれば、
売上は設定しているものより1.2倍に増える。
```

#### 削除する仕様

以下の仕様は削除する。

```text
SNS投稿が3日以下であれば、売上が半分程度になる。
```

つまり、SNS投稿が3日以下でもペナルティは発生しない。

#### 画面表示の方針

SNSによる1.2倍効果は、**画面には倍率として表示しない**。

表示しない例：

```text
SNS効果：1.2倍
売上が120%になります
SNS投稿4日以上なので売上1.2倍
```

表示してよい例：

```text
これまでのSNS発信が集客に良い影響を与えました。
```

または、結果だけを表示する。

```text
本日の売上：〇〇円
本日の利益：〇〇円
```

学生には、内部的に1.2倍がかかったことを明示しない。

ただし、履歴画面では「SNS投稿数」は表示してよい。

### 1.5 学園祭イベントの表示方針

学園祭では需要が変わる、という説明はしてよい。

ただし、画面上では具体的な倍率・パーセントを表示しない。

表示してはいけない例：

```text
需要が120%になります
需要が1.2倍になります
売上が70%になります
```

表示してよい例：

```text
学園祭のため、いつもより来店が増えそうです。
他の出店も多いため、商品や価格の工夫が大切です。
```

内部計算ではイベント係数を使ってよいが、学生向け画面には倍率やパーセントを出さない。

### 1.6 スタート画面の入力欄改善

最初の画面の以下の入力欄を大きく表示する。

- チーム名
- 店舗名

小さなテキストボックスではなく、PC画面で見やすく、入力しやすいサイズにする。

### 1.7 仕入単価上昇イベント

3日目に仕入単価が上がる。

ただし、3日目だけではなく、**3日目以降ずっと仕入単価が高い状態が続く**。

```text
1〜2日目：通常仕入単価
3〜7日目：仕入単価1.2倍
```

## 2. アプリ概要

### アプリ名

学内カフェ店長ゲーム

### 利用目的

大学の授業で、学生が2人1組になり、学内カフェの店長として、仕入れ、価格設定、SNS投稿、販売、在庫、廃棄、利益を体験的に学ぶ。

### 学習目的

学生に以下を体験させる。

- 売上と利益の違い
- 仕入れすぎによる在庫・廃棄リスク
- 価格設定による需要変化
- SNS投稿の継続が後半の集客に影響する場合があること
- 学園祭、仕入単価上昇など、外部環境変化への対応
- 商品ごとに在庫期限が違うこと
- 過去の結果を見て、次の仕入れや価格を考えること

## 3. 開発方針

### 技術構成

以下で実装する。

- React
- Vite
- JavaScript または TypeScript
- LocalStorage

TypeScriptが使いやすければTypeScriptでよい。迷う場合はJavaScriptでよい。

### 実行方法

以下で動作するようにする。

```bash
npm install
npm run dev
```

### データベース

初期版ではデータベースは使わない。

すべてブラウザ内で完結させる。

途中データはLocalStorageに保存する。

### 対象端末

PCブラウザ向け。

スマートフォン最適化は不要。

### UI方針

- 白背景
- 大きめの文字
- カード型レイアウト
- 数値が見やすい
- 入力欄は大きくする
- 1画面あたりの操作を少なくする
- 授業中に説明しやすい
- 7日間でテンポよく進められる

## 6. 仕入単価上昇ルール

3日目以降、全商品の仕入単価が1.2倍になる。

重要：3日目だけではない。**3日目から7日目まで継続する。**

| 商品 | 1〜2日目の仕入単価 | 3〜7日目の仕入単価 |
|---|---:|---:|
| コーヒー | 30円 | 36円 |
| サンドイッチ | 180円 | 216円 |
| クッキー | 150円 | 180円 |

### 実装ルール

```text
if currentDay >= 3:
  actualCostPrice = baseCostPrice * 1.2
else:
  actualCostPrice = baseCostPrice
```

画面には、3日目以降は以下のように表示する。

```text
仕入単価が上昇しています。仕入れ数量と販売価格に注意してください。
```

## 8. SNS投稿ルール

SNS投稿は費用0円。

毎日「投稿する / 投稿しない」を選べる。

### 8.1 SNS投稿履歴

SNS投稿の有無を日別に保存する。

```json
{
  "day": 1,
  "snsPosted": true
}
```

### 8.2 1〜5日目のSNS投稿数

6日目の営業結果を計算するときに、1〜5日目のSNS投稿数を数える。

```text
snsCountDays1to5 = 1日目〜5日目のSNS投稿回数
```

### 8.3 6日目のSNS効果

6日目は、これまでのSNS投稿が学園祭集客に影響する。

```text
snsCountDays1to5 >= 4 の場合：
  6日目の売上を、計算後に1.2倍する
```

重要：

- この1.2倍は、**6日目の売上に対して適用**する。
- 需要に掛けるのではなく、最終的に計算された売上に掛ける。
- 売上1.2倍に合わせて、本日の利益と現在資金も再計算する。
- 販売数は変えなくてよい。結果として「SNSにより客単価や追加購入が増えた」と考える。
- 画面には「1.2倍」「120%」などの数値は表示しない。

### 8.4 3日以下の場合

SNS投稿が3日以下の場合でも、ペナルティは発生しない。

削除する仕様：

```text
SNS投稿が3日以下なら半分程度になる
```

実装してはいけない処理：

```text
if snsCountDays1to5 <= 3:
  売上を0.5倍する
```

### 8.5 6日目の画面表示

6日目にSNS投稿4日以上の効果が発生した場合、学生向けには以下のように表示する。

```text
これまでのSNS発信が集客に良い影響を与えました。
```

または、コメントを出さず、結果だけ表示してもよい。

表示してはいけない文言：

```text
SNS効果：1.2倍
売上が120%になりました
SNS投稿が4日以上なので売上1.2倍
```

### 8.6 その他の日のSNS効果

初期版では、6日目以外のSNS効果は小さくする。

推奨実装：

```text
SNS投稿あり：SNS係数 = 1.00〜1.10のランダム
SNS投稿なし：SNS係数 = 1.00
```

ただし、画面には具体的な倍率を表示しない。

表示例：

```text
SNS投稿をしました。少し反応があったようです。
```

```text
SNS投稿をしましたが、今日は大きな反応はありませんでした。
```

```text
今日はSNS投稿をしませんでした。
```

## 9. 需要計算ロジック

### 基本式

商品ごとの需要は以下で計算する。

```text
予測需要 = 基本需要 × 価格係数 × イベント係数 × SNS係数 × ランダム係数
販売数 = min(販売可能在庫数, 予測需要)
```

小数点以下は切り捨てる。

```text
販売数 = Math.floor(販売数)
```

### ランダム係数

価格・イベント・SNSを反映した後、最後にランダム係数を掛ける。

```text
ランダム係数 = 0.85〜1.15
```

つまり、プラス・マイナス15%の範囲で需要が変動する。

ランダム係数は、営業開始時に決定し、営業結果画面に表示してよい。

ただし、学園祭イベントや6日目のSNS売上効果については、具体的な倍率・パーセントを学生向けに表示しない。

### 6日目の売上補正

6日目のみ、SNS投稿数条件を満たした場合に、商品別売上または本日の売上に補正をかける。

推奨は、本日の売上合計に補正をかけること。

```text
通常計算：
本日の売上 = 全商品の売上合計
本日の利益 = 本日の売上 - 本日の仕入原価
現在資金 = 前日の資金 - 本日の仕入原価 + 本日の売上

6日目かつ1〜5日目SNS投稿4日以上の場合：
補正後売上 = 本日の売上 × 1.2
補正後利益 = 補正後売上 - 本日の仕入原価
補正後現在資金 = 前日の資金 - 本日の仕入原価 + 補正後売上
```

画面では補正後の売上・利益・資金のみを表示する。

## 10. 価格による需要変化

価格条件ごとに価格係数を設定する。

| 価格条件 | 価格係数 |
|---|---:|
| 標準価格と同じ | 1.00 |
| 標準価格より10%安い | 1.20 |
| 標準価格より20%安い | 1.40 |
| 標準価格より10%高い | 0.85 |
| 標準価格より20%高い | 0.70 |
| 標準価格より30%以上高い | 0.50 |

実装では、販売価格と標準販売価格の比率から近い係数を適用する。

### 実装例

```text
priceRatio = sellingPrice / standardPrice

priceRatio <= 0.80 → 1.40
priceRatio <= 0.90 → 1.20
priceRatio <= 1.00 → 1.00
priceRatio <= 1.10 → 0.85
priceRatio <= 1.20 → 0.70
priceRatio >  1.20 → 0.50
```

## 12. 在庫・廃棄ルール

### コーヒー

コーヒーは賞味期限なしとして扱う。

売れ残りはすべて翌日以降に持ち越す。

```text
コーヒー販売可能在庫 = 前日までのコーヒー在庫 + 今日の仕入数
コーヒー翌日在庫 = コーヒー販売可能在庫 - コーヒー販売数
```

### サンドイッチ

サンドイッチは当日廃棄。

```text
サンドイッチ販売可能在庫 = 今日の仕入数
サンドイッチ廃棄数 = サンドイッチ販売可能在庫 - サンドイッチ販売数
翌日在庫 = 0
```

### クッキー

クッキーは、仕入れた日と翌日まで販売可能。

翌々日には廃棄する。

クッキーは仕入日別に在庫を管理する。

```text
クッキー販売可能在庫 = 前日からの持ち越しクッキー + 今日仕入れたクッキー
```

販売時は古い在庫から先に売る。

営業終了後：

```text
前日から持ち越したクッキーの売れ残り → 廃棄
今日仕入れたクッキーの売れ残り → 翌日に持ち越し
```

## 13. 利益計算

### 商品別売上

```text
商品別売上 = 販売価格 × 販売数
```

### 商品別仕入原価

```text
商品別仕入原価 = 当日の仕入単価 × 今日の仕入数
```

### 本日の売上

```text
本日の売上 = 全商品の売上合計
```

ただし、6日目かつ1〜5日目SNS投稿4日以上の場合は、補正後売上を本日の売上として扱う。

### 本日の仕入原価

```text
本日の仕入原価 = 全商品の当日仕入原価合計
```

### 本日の利益

```text
本日の利益 = 本日の売上 - 本日の仕入原価
```

### 現在資金

```text
現在資金 = 前日の資金 - 本日の仕入原価 + 本日の売上
```

### 注意

会計上の厳密な在庫評価は初期版では行わない。

コーヒーとクッキーの在庫は、翌日以降の販売可能在庫としてのみ管理する。

## 19. 初期版の完成条件

以下を満たせば完成とする。

- 7日間のゲームが進行する
- 月曜日から始まり、土日の学園祭を含む
- 初期資金50,000円で開始する
- 基本ニーズ100人をもとに需要計算する
- 営業時間10:00〜17:00を表示する
- 3日目以降、仕入単価が1.2倍になり、その状態が7日目まで続く
- 5日目は学園祭準備として扱う
- 6日目・7日目は学園祭として扱う
- 6日目に、1〜5日目のSNS投稿が4日以上なら売上に1.2倍補正をかける
- SNS投稿3日以下でも売上半減などのペナルティは実装しない
- 6日目のSNS売上補正について、学生向け画面に1.2倍・120%などの具体的数値を表示しない
- 学園祭の需要変化について、学生向け画面に具体的な倍率やパーセントを表示しない
- コーヒーの在庫が翌日以降に持ち越される
- サンドイッチは当日廃棄される
- クッキーは翌日まで販売でき、翌々日には廃棄される
- いちごトッピングは存在しない
- 価格係数が需要に反映される
- ランダム係数により需要が±15%変動する
- 履歴画面で過去の日別結果を確認できる
- LocalStorageで途中保存・再開できる
- 最終結果画面で結果をコピーできる
- スタート画面のチーム名・店舗名入力欄が大きく表示される
