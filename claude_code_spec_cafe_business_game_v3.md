# Claude Code向け仕様書：学内カフェ経営ゲーム v3

## 0. この仕様書の目的

このMarkdownは、大学授業向けの「学内カフェ経営ゲーム」を、Claude Codeで実装するための開発指示書である。

既存仕様からの主な変更点は以下である。

- ゲーム期間を5日間から10日間に変更する
- 月曜日から開始し、土日を含む10日間の営業にする
- 6日目・7日目を学園祭イベントにする
- 基本ニーズを100人にする
- 初期資金を50,000円にする
- カフェ営業時間を10:00〜17:00にする
- コーヒーの仕入単価・販売価格を変更する
- スイーツを「焼き菓子」と「いちごトッピング」に分ける
- 焼き菓子は翌日まで販売可能にする
- いちごトッピングは当日廃棄にする
- SNS投稿は即時効果よりも、後半イベントへの累積効果を重視する
- 3日目、5日目、6日目、8日目、9日目、10日目に特殊イベントを発生させる
- 価格係数・イベント係数・SNS係数を反映した後、需要に±15%のランダム変動を加える
- LocalStorageで途中保存・再開できるようにする

この仕様では、特に以下の変更内容を重要実装項目として扱う。

1. 10日間運営
2. 学園祭イベント
3. SNS累積効果
4. 在庫・廃棄ルールの違い
5. 焼き菓子といちごトッピングの関係
6. ランダム需要変動
7. 競合店・仕入単価上昇・学園祭疲れなどのイベント処理

---

## 1. アプリ概要

### アプリ名

学内カフェ店長ゲーム

### 利用目的

大学の授業で、学生が2人1組になり、学内カフェの店長として、仕入れ、価格設定、SNS投稿、販売、在庫、廃棄、利益を体験的に学ぶ。

### 学習目的

学生に以下を体験させる。

- 売上と利益の違い
- 仕入れすぎによる在庫・廃棄リスク
- 価格設定による需要変化
- SNSのような事前活動が後半の集客に影響すること
- 学園祭、競合店、仕入単価上昇など外部環境変化への対応
- 商品ごとに在庫期限が違うこと
- 売れそうな日と売れにくい日を見越した仕入れ判断

## 2. 開発方針

### 技術構成

以下で実装する。

- React
- Vite
- JavaScript または TypeScript
- LocalStorage

TypeScriptが使いやすければTypeScriptでよい。迷う場合はJavaScriptでよい。

### 実行方法

以下で動作するようにする。

```bash
npm install
npm run dev
```

### データベース

初期版ではデータベースは使わない。

すべてブラウザ内で完結させる。

途中データはLocalStorageに保存する。

### 対象端末

PCブラウザ向け。

スマートフォン最適化は不要。

### UI方針

- 白背景
- 大きめの文字
- カード型レイアウト
- 数値が見やすい
- 1画面あたりの操作を少なくする
- 授業中に説明しやすい
- 10日間でもテンポよく進められる

## 3. 基本設定

| 項目 | 内容 |
|---|---|
| プレイ人数 | 2人1組 |
| ゲーム期間 | 10日間 |
| 開始曜日 | 月曜日 |
| 営業時間 | 10:00〜17:00 |
| 基本ニーズ | 通常日は100人 |
| 初期資金 | 50,000円 |
| 保存方式 | LocalStorage |
| 目的 | 最終資金・総利益を高める |

### 初期資金

初期資金は50,000円とする。

理由：

- 基本ニーズを100人にしたため、従来より仕入額が増える
- 学園祭前後に追加仕入れできる余力が必要
- 仕入れすぎや価格設定ミスがあってもゲームが継続できるようにする

表示上は以下のように説明してもよい。

```text
初期資金：50,000円
内訳イメージ：2人の出資金20,000円 + 借入金30,000円
```

ただし、会計上の借入返済処理は初期版では行わない。

## 4. 商品仕様

商品は以下の4種類とする。

| 商品 | 仕入単価 | 標準販売価格 | 標準需要 | 在庫・廃棄ルール |
|---|---:|---:|---:|---|
| コーヒー | 30円 | 100円 | 60個 | 賞味期限なし。売れ残りは翌日以降も販売可能 |
| サンドイッチ | 180円 | 320円 | 35個 | その日の終わりに廃棄 |
| 焼き菓子 | 150円 | 300円 | 25個 | 仕入れた日と翌日まで販売可能。翌々日には廃棄 |
| いちごトッピング | 30円 | 100円 | 20個 | その日の終わりに廃棄。焼き菓子販売数を上限に販売される |

### 重要：スイーツの扱い

旧仕様の「スイーツ」は、v3では以下の2つに分けて扱う。

- 焼き菓子
- いちごトッピング

焼き菓子は単体でも販売可能。

いちごトッピングは、焼き菓子を購入した顧客にだけ追加販売される。

したがって、いちごトッピングの販売数は、焼き菓子販売数を超えてはいけない。

```text
いちごトッピング販売数 = min(焼き菓子販売数, いちご需要, いちご在庫)
```

## 5. 10日間のイベント設計

10日間のイベントは、配列または設定ファイルのような形で管理する。

### イベント一覧

| 日数 | 曜日 | イベント | 需要・売上への影響 | 市場メモ |
|---:|---|---|---|---|
| 1日目 | 月 | 通常営業 | 基本ニーズ100人 | まずは標準的な需要が見込まれる |
| 2日目 | 火 | 通常営業 | 基本ニーズ100人 | 前日の結果を見て仕入れと価格を調整する |
| 3日目 | 水 | 仕入単価上昇 | 全商品の仕入単価が1.2倍 | 仕入れコストが上がるため、価格と数量の判断が重要 |
| 4日目 | 木 | 通常営業 | 基本ニーズ100人 | 学園祭前にSNSを継続するか判断する |
| 5日目 | 金 | 学園祭準備・競合店出店 | 競合店の影響で需要係数0.7 | 学園祭準備で人はいるが、競合店に一部流れる |
| 6日目 | 土 | 学園祭1日目 | 1〜5日目のSNS投稿日数で大きく変化 | 他店舗も多く出店。これまでのSNS発信が集客に影響 |
| 7日目 | 日 | 学園祭2日目 | 学園祭需要は高め。SNS効果の余韻を反映 | 学園祭2日目。前日の評判も影響する |
| 8日目 | 月 | 通常営業・学園祭疲れ | ニーズ70%。イベント係数0.7 | 翌週月曜日で授業は普段通りだが、学園祭疲れで来店が少ない |
| 9日目 | 火 | ゼミ発表会 | サンドイッチ需要1.3倍。SNS疲れ条件あり | 発表会前後の軽食需要が高まる |
| 10日目 | 水 | サークルイベント | コーヒー1.2倍、焼き菓子・いちご1.1倍。SNS疲れ条件あり | イベント参加者の休憩需要がある |

### 5日目の競合店イベント

5日目は、競合店ができる。

仕様上は「売上が70%になる」と表現されているが、実装では需要係数0.7として扱う。

```text
5日目イベント係数 = 0.7
```

その結果、販売数量と売上が概ね70%になる。

## 6. SNS投稿ルール

SNS投稿は費用0円。

SNSは、毎日「投稿する / 投稿しない」を選べる。

### 基本方針

v3では、SNS投稿は単日のランダム効果ではなく、主に後半イベントに効く累積効果として扱う。

特に6日目の学園祭1日目に大きく影響する。

### SNSルール一覧

| 期間 | SNSの扱い |
|---|---|
| 1〜5日目 | SNS投稿の有無を記録する。通常日は需要への直接効果は小さめ、または効果なし |
| 6日目 | 1〜5日目のうちSNS投稿が4日以上なら需要1.2倍。3日以下なら需要0.5倍 |
| 7日目 | 6日目の結果を受け、SNS投稿継続なら需要1.1倍程度の余韻効果 |
| 8〜10日目 | 投稿した日はランダムで需要100〜115%程度 |
| 特殊条件 | 1〜8日目まで8日連続でSNS投稿していた場合、発信疲れにより9日目の売上70%、10日目の売上80% |

### 実装詳細

#### 1〜5日目

SNS投稿回数をカウントする。

1〜5日目については、初期版では即時効果はなしでよい。

```text
1〜5日目SNS係数 = 1.0
```

ただし、後で拡張しやすいように、SNS投稿有無は日別履歴に保存する。

#### 6日目：学園祭1日目

1〜5日目のSNS投稿回数を数える。

```text
snsCountDays1to5 = 1日目〜5日目のSNS投稿回数
```

判定：

```text
snsCountDays1to5 >= 4 の場合：SNS係数 = 1.2
snsCountDays1to5 <= 3 の場合：SNS係数 = 0.5
```

#### 7日目：学園祭2日目

7日目にSNS投稿をする、または6日目にもSNS投稿していた場合は、余韻効果としてSNS係数1.1を与える。

実装を単純にする場合は以下でよい。

```text
7日目にSNS投稿する場合：SNS係数 = 1.1
7日目にSNS投稿しない場合：SNS係数 = 1.0
```

#### 8〜10日目：通常SNS効果

8〜10日目は、SNS投稿した日にだけ軽いランダム効果を与える。

```text
SNS投稿あり：SNS係数 = 1.00〜1.15 のランダム
SNS投稿なし：SNS係数 = 1.00
```

#### 8日連続SNS投稿による発信疲れ

1日目から8日目まで、8日連続でSNS投稿していた場合、9日目と10日目にマイナス効果を発生させる。

```text
1〜8日目がすべてSNS投稿ありの場合：
  9日目：SNS疲れ係数 = 0.7
  10日目：SNS疲れ係数 = 0.8
```

この場合、9日目・10日目の通常SNS係数やイベント係数と掛け合わせてよい。

画面には以下のようなコメントを表示する。

```text
SNSを連日発信しすぎたため、発信疲れによるマイナス反応が出ました。
```

## 7. 需要計算ロジック

### 基本式

商品ごとの需要は以下で計算する。

```text
予測需要 = 基本需要 × 価格係数 × イベント係数 × SNS係数 × ランダム係数
販売数 = min(販売可能在庫数, 予測需要)
```

小数点以下は切り捨てる。

```text
販売数 = Math.floor(販売数)
```

### ランダム係数

価格・イベント・SNSを反映した後、最後にランダム係数を掛ける。

```text
ランダム係数 = 0.85〜1.15
```

つまり、プラス・マイナス15%の範囲で需要が変動する。

ランダム係数は、営業開始時に決定し、営業結果画面に表示する。

例：

```text
ランダム需要変動：+8%
ランダム需要変動：-12%
```

### 注意

ランダム係数は、毎回結果が変わる要素である。

ただし、営業結果が確定した後は、その日のランダム係数をLocalStorageに保存し、再開しても同じ結果を表示する。

## 8. 価格による需要変化

価格条件ごとに価格係数を設定する。

| 価格条件 | 価格係数 |
|---|---:|
| 標準価格と同じ | 1.00 |
| 標準価格より10%安い | 1.20 |
| 標準価格より20%安い | 1.40 |
| 標準価格より10%高い | 0.85 |
| 標準価格より20%高い | 0.70 |
| 標準価格より30%以上高い | 0.50 |

実装では、販売価格と標準販売価格の比率から近い係数を適用する。

### 実装例

```text
priceRatio = sellingPrice / standardPrice

priceRatio <= 0.80 → 1.40
priceRatio <= 0.90 → 1.20
priceRatio <= 1.00 → 1.00
priceRatio <= 1.10 → 0.85
priceRatio <= 1.20 → 0.70
priceRatio >  1.20 → 0.50
```

## 9. イベント係数の詳細

### 3日目：仕入単価上昇

3日目は全商品の仕入単価が1.2倍になる。

| 商品 | 通常仕入単価 | 3日目仕入単価 |
|---|---:|---:|
| コーヒー | 30円 | 36円 |
| サンドイッチ | 180円 | 216円 |
| 焼き菓子 | 150円 | 180円 |
| いちごトッピング | 30円 | 36円 |

3日目だけ、仕入原価計算に上昇後の単価を使う。

### 5日目：競合店

5日目は競合店ができる。

```text
全商品イベント係数 = 0.7
```

### 6日目：学園祭1日目

6日目はSNS投稿累積効果が強く出る。

```text
1〜5日目SNS投稿4日以上 → SNS係数 = 1.2
1〜5日目SNS投稿3日以下 → SNS係数 = 0.5
```

イベントとしての基本需要増加を入れる場合は、学園祭係数1.2を別途設定してもよい。

ただし、初期版ではSNS効果だけでもよい。

### 7日目：学園祭2日目

学園祭2日目。

初期版では以下でよい。

```text
全商品イベント係数 = 1.1
```

さらにSNS投稿を継続していればSNS係数1.1を与える。

### 8日目：学園祭疲れ

8日目は、学園祭疲れで学生が普段の70%しか来ない。

```text
全商品イベント係数 = 0.7
```

### 9日目：ゼミ発表会

9日目はゼミ発表会がある。

```text
サンドイッチイベント係数 = 1.3
その他商品イベント係数 = 1.0
```

ただし、1〜8日目までSNSを8日連続で発信していた場合は、発信疲れの影響を追加する。

```text
SNS疲れ係数 = 0.7
```

### 10日目：サークルイベント

10日目はサークルイベントがある。

```text
コーヒーイベント係数 = 1.2
焼き菓子イベント係数 = 1.1
いちごトッピングイベント係数 = 1.1
サンドイッチイベント係数 = 1.0
```

ただし、1〜8日目までSNSを8日連続で発信していた場合は、発信疲れの影響を追加する。

```text
SNS疲れ係数 = 0.8
```

## 10. 在庫・廃棄ルール

### コーヒー

コーヒーは賞味期限なしとして扱う。

売れ残りはすべて翌日以降に持ち越す。

```text
コーヒー販売可能在庫 = 前日までのコーヒー在庫 + 今日の仕入数
コーヒー翌日在庫 = コーヒー販売可能在庫 - コーヒー販売数
```

### サンドイッチ

サンドイッチは当日廃棄。

```text
サンドイッチ販売可能在庫 = 今日の仕入数
サンドイッチ廃棄数 = サンドイッチ販売可能在庫 - サンドイッチ販売数
翌日在庫 = 0
```

### 焼き菓子

焼き菓子は、仕入れた日と翌日まで販売可能。

翌々日には廃棄する。

焼き菓子は仕入日別に在庫を管理する。

最低限、以下の2つを持てばよい。

```text
bakeryStockTodayPurchase
bakeryStockYesterdayCarryover
```

営業時の販売可能在庫：

```text
焼き菓子販売可能在庫 = 前日からの持ち越し焼き菓子 + 今日仕入れた焼き菓子
```

販売時は、古い在庫から先に売る。

```text
1. 前日からの持ち越し在庫を先に販売
2. 足りなければ今日仕入れた在庫を販売
```

営業終了後：

```text
前日から持ち越した焼き菓子の売れ残り → 廃棄
今日仕入れた焼き菓子の売れ残り → 翌日に持ち越し
```

### いちごトッピング

いちごトッピングは当日廃棄。

焼き菓子を買った人にだけ売れる。

```text
いちご販売可能在庫 = 今日の仕入数
いちご販売数 = min(焼き菓子販売数, いちご需要, いちご販売可能在庫)
いちご廃棄数 = いちご販売可能在庫 - いちご販売数
翌日在庫 = 0
```

## 11. 利益計算

### 商品別売上

```text
商品別売上 = 販売価格 × 販売数
```

### 商品別仕入原価

```text
商品別仕入原価 = 当日の仕入単価 × 今日の仕入数
```

### 本日の売上

```text
本日の売上 = 全商品の売上合計
```

### 本日の仕入原価

```text
本日の仕入原価 = 全商品の当日仕入原価合計
```

### 本日の利益

```text
本日の利益 = 本日の売上 - 本日の仕入原価
```

### 現在資金

```text
現在資金 = 前日の資金 - 本日の仕入原価 + 本日の売上
```

### 注意

会計上の厳密な在庫評価は初期版では行わない。

コーヒーと焼き菓子の在庫は、翌日以降の販売可能在庫としてのみ管理する。

## 13. LocalStorage保存仕様

### 保存キー

cafe_business_game_v3_state

### 保存するデータ

最低限、以下を保存する。

```json

  "coffeeStock": 0,
  "bakeryCarryoverStock": 0,
  "snsHistory": [],
  "dailyInputs": [],
  "dailyResults": [],
  "reflectionNotes": [],
  "isFinished": false
}
```

### 追加で保存するデータ

- 曜日情報
- イベント情報
- コーヒーの持ち越し在庫
- 焼き菓子の仕入日別在庫
- いちごトッピングの当日仕入数・販売数・廃棄数
- SNS投稿日数
- イベント係数
- 価格係数
- SNS係数
- ランダム係数
- 競合店の影響
- 仕入単価上昇の反映結果
- 最終結果

### 保存タイミング

- ゲーム開始時
- 仕入れ・価格設定を確定した時
- 営業結果が確定した時
- 振り返りを入力した時
- 次の日へ進む時
- 最終結果が表示された時

### 注意

営業結果確定後は、ランダム係数も保存する。

再開したときに、同じ日の結果が変わってはいけない。

## 14. 入力チェック

### 必須チェック

- チーム名が空ならゲーム開始できない
- 店舗名が空ならゲーム開始できない
